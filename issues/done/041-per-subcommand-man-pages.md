# Per-subcommand man pages — `bitcoin-<verb>(1)` for every shipping command

## Description

**As a** user running `man bitcoin-tx` or `man bitcoin-bip174`
**I want** a man page that's actually there — and that matches
what `bitcoin <verb> --help` tells me
**So that** the documentation surface for `bitcoin` matches the
Unix conventions a wallet operator already knows from
`git-status(1)`, `systemd.service(5)`, etc., and I never have
to grep source to learn what a verb does.

`bitcoin(1)` already exists (FEAT-015 partial) and covers the
overall dispatcher. This feature codifies the **convention** for
per-subcommand pages and lands the first batch alongside
FEAT-035 / 036 / 037 / 038 in 1.23.0. Every future verb (1.24.0
Sparrow-flex, 1.25.0 tax, 1.26.0 Taproot) inherits the same
convention.

## Convention

### Format

groff `-man` macros (`.TH`, `.SH`, `.IP`), matching the
existing `share/man/man1/bitcoin.1`. Not mdoc — keep the
existing format so a single `man` reader renders everything
consistently and the Makefile install rule stays a one-line
glob.

### File naming

| Verb kind            | Source path                          | Renders as       |
|----------------------|---------------------------------------|------------------|
| Parent dispatcher    | `share/man/man1/bitcoin.1`            | `bitcoin(1)`     |
| BIP plugin           | `share/man/man1/bitcoin-bip174.1`     | `bitcoin-bip174(1)` |
| Object verb          | `share/man/man1/bitcoin-tx.1`         | `bitcoin-tx(1)`  |
| Deprecated alias     | `share/man/man1/bitcoin-psbt.1`       | `bitcoin-psbt(1)` (one-line `.so bitcoin-bip174.1`) |

### Required sections (per page)

Every `bitcoin-<verb>(1)` page has, at minimum:

1. `.SH NAME` — `bitcoin-<verb> \- one-line summary`
2. `.SH SYNOPSIS` — every form of the verb's invocation
3. `.SH DESCRIPTION` — what the verb does, in plain prose
4. `.SH OPTIONS` — every flag with its semantics
5. `.SH EXIT STATUS` — what each non-zero exit means
6. `.SH FILES` — paths the verb reads/writes (wallets, cache,
   config) — informative for operators and required for tax
   reproducibility (FEAT-039)
7. `.SH ENVIRONMENT` — env vars the verb reads (if any)
8. `.SH STANDARDS` — for `bipXXX` plugins, the BIP citation
   with year and link; for object verbs, omit
9. `.SH SEE ALSO` — sibling pages (e.g. `bitcoin-tx(1)` →
   `bitcoin-bip174(1)`, `bitcoin-utxo(1)`, `bitcoin-wallet(1)`)
10. `.SH HISTORY` — release the verb was introduced in (and,
    for deprecated aliases, the release that introduced the
    canonical name and the release the alias is removed)

### Cross-referencing from `bitcoin(1)`

The top-level `bitcoin(1)` gets a new `.SH COMMANDS` section
that lists every shipping verb with a one-line summary and an
in-page `.BR bitcoin-<verb> (1)` cross-reference. Generated by
hand — the file is short enough that drift is caught at review
time.

### Deprecated-alias pages

Every deprecated alias ships a one-line man page that `.so`-
includes the canonical:

    .\" bitcoin-psbt(1) — deprecated alias for bitcoin-bip174(1)
    .so man1/bitcoin-bip174.1

`man bitcoin-psbt` renders the canonical page. The page itself
mentions the alias in its `HISTORY` section.

### Help-text consistency

`bitcoin <verb> --help` must list the same flags, in the same
order, that `bitcoin-<verb>(1)` documents in `.SH OPTIONS`.
This isn't a hard test (output formats differ) but a review
checklist item per `skills/features.md`.

### Install

No Makefile change needed — `Makefile:59-60` already globs
`share/man/man1/*` into `$BUILD_DIR$MANDIR/man1/`. New `.1`
files in the source dir are picked up automatically.

### Generation

Man pages are **hand-written**, not generated from `--help`
output. The output of `--help` is terse; man pages are
discursive. Two artifacts, two voices.

## Implementation

PR 1 of FEAT-041 (lands alongside FEAT-035 in 1.23.0):

- New `share/man/man1/bitcoin-bip13.1`, `bitcoin-bip32.1`,
  `bitcoin-bip39.1`, `bitcoin-bip173.1`, `bitcoin-bip174.1`,
  `bitcoin-bip350.1`, `bitcoin-bip380.1`, `bitcoin-wif.1`,
  `bitcoin-daemon.1`, `bitcoin-backend.1`, `bitcoin-wallet.1`.
- Deprecated-alias pages: `bitcoin-psbt.1`, `bitcoin-descriptor.1`,
  `bitcoin-mnemonic-to-seed.1`. Each is a one-line `.so` into
  its canonical.
- Update `share/man/man1/bitcoin.1`: add `.SH COMMANDS` with
  cross-references.

PRs 3 and 4 of the milestone (FEAT-036 `tx`, FEAT-037 `utxo`)
each add their own page in the same PR that introduces the
verb. Same convention; same checklist.

FEAT-038 (tax-label vocabulary) updates `bitcoin-wallet(1)` to
document `label --tax` and the closed taxonomy.

FEAT-039 / 040 (1.25.0) ship `bitcoin-tax.1` and
`bitcoin-price.1` when those verbs land.

## Regression protection

New bats cases (one per page):

- `man -w bitcoin-<verb>` resolves to a path under
  `$BUILD_DIR$MANDIR` after `make install`.
- `man bitcoin-<verb>` exits 0 and emits a non-empty page.
- Every page's `.SH NAME` line matches the format `bitcoin-<verb>
  \- ...` (one-line check via grep).
- Deprecated-alias pages render the canonical page's content
  (grep the canonical's `.SH SYNOPSIS` after rendering the
  alias).
- Top-level `bitcoin(1)` `.SH COMMANDS` mentions every shipping
  verb (cross-check against `bitcoin modules` output).

A single bats helper, `tests/unit/manpages.bats`, drives all
of the above against the installed `$MANDIR`.

## Acceptance criteria

1. Every verb listed in `bitcoin modules` output has a
   corresponding `share/man/man1/bitcoin-<verb>.1` source file.
2. Every page conforms to the 10-section structure above (NAME,
   SYNOPSIS, DESCRIPTION, OPTIONS, EXIT STATUS, FILES,
   ENVIRONMENT, STANDARDS [BIP plugins only], SEE ALSO,
   HISTORY).
3. Deprecated aliases ship as `.so`-include one-liners; `man
   bitcoin-<alias>` renders the canonical page.
4. `bitcoin(1)` `.SH COMMANDS` cross-references every shipping
   verb.
5. `make install` puts every page under `$MANDIR/man1/` (no
   Makefile change needed beyond the existing glob).
6. `tests/unit/manpages.bats` covers the assertions above and
   runs in the unit tier.
7. The "help-text consistency" review checklist line is added
   to `skills/features.md` so every future feature lands its
   man page with the verb itself.

## Why this is a separate feature

The convention applies forever and across milestones, so it
gets its own issue rather than being a footnote in FEAT-035.
But the **first batch** of pages must land with FEAT-035 — you
can't rename `psbt` to `bip174` without renaming the page at
the same time. FEAT-041 PR 1 and FEAT-035 PR 1 are the same
PR in practice.

## Relationship to FEAT-015

FEAT-015 (`bitcoin(1)` man page, bash completion, README
walkthrough) shipped the parent `bitcoin(1)` and the
walkthrough at `docs/bitcoin-walkthrough.md`. FEAT-041 extends
that surface to **per-subcommand** pages and codifies the
convention going forward. Bash completion stays in FEAT-015's
scope (a future PR there can derive completions from the
SYNOPSIS lines once the convention is settled).
